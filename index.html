Week 01

Introduction to Hacking Lab. Offensive and Defensive Security. Penetration Testing Stages

Instructor: Senior Lecturer, MSc Aliya Abdiraman

A. Lecture Overview

This foundational lecture establishes the conceptual bedrock of the entire Hacking Lab course. Students are introduced to the binary world of cybersecurity — the attackers who probe systems for weakness, and the defenders who harden them against intrusion. The lecture maps the ethical and methodological boundaries between offensive security (penetration testing, red teaming, ethical hacking) and defensive security (blue teaming, incident response, threat hunting). Students also receive their first exposure to Kali Linux, the industry-standard penetration testing distribution, and learn basic terminal navigation essential for all subsequent lab work.

  • Understand the terminology distinguishing ethical hackers from malicious actors
  • Differentiate clearly between offensive and defensive security disciplines
  • Map the six canonical stages of a penetration testing engagement
  • Install and navigate Kali Linux in a virtualized environment

B. Core Concepts & Definitions

Hacker

A person who explores, analyzes, and manipulates computer systems, networks, or applications — often to discover vulnerabilities. The term is morally neutral; intent determines whether the activity is constructive or destructive.

White Hat Hacker

An ethical hacker who uses offensive techniques defensively — always with explicit permission from system owners. White hats work to identify and remediate vulnerabilities before malicious actors can exploit them.

Black Hat Hacker

A malicious actor who intrudes into systems without authorization and with harmful or criminal intent — data theft, sabotage, financial fraud, or ransom. Black hat activity is illegal in virtually every jurisdiction.

Gray Hat Hacker

A hacker who operates without prior authorization but without explicitly malicious intent — often disclosing vulnerabilities after discovering them, sometimes demanding a fee for remediation assistance. Gray hat activity occupies a legal and ethical grey zone.

Vulnerability

A weakness or flaw in a system, application, network, or process that could be exploited to violate security policy — compromising confidentiality, integrity, or availability.

Exploit

A method, code, or technique that takes advantage of a specific vulnerability to achieve an unauthorized effect — such as gaining access, escalating privileges, or causing denial of service.

Penetration Testing

A structured, authorized simulation of a real-world attack against a target system or network. The goal is to identify exploitable vulnerabilities before malicious actors do, and to provide remediation guidance.

Red Team

A group of specialists who conduct long-duration, realistic simulations of advanced persistent threats (APTs) against an organization's full infrastructure — including physical, social, and technical attack vectors.

Blue Team

The defensive security team responsible for protecting infrastructure, monitoring for threats, analyzing alerts, and responding to incidents. Blue teamers operate Security Operations Centers (SOCs).

C. Expanded Academic Notes

⚠ Offensive Security

Offensive security is a proactive discipline that simulates attacker behavior to identify weaknesses before real attackers exploit them. Core activities include:

  • Penetration Testing — time-boxed, scoped simulated attacks on systems with full authorization
  • Red Teaming — long-duration adversarial simulations replicating nation-state or APT actors
  • Vulnerability Research — discovering novel (zero-day) or known vulnerabilities in software
  • Ethical Hacking — using the full toolkit of malicious techniques, lawfully, to harden systems

Real Example: Before a major bank launches a new online portal, it commissions a penetration testing firm to attempt to breach it using the same techniques a criminal would. Vulnerabilities found are patched before launch.

🛡 Defensive Security

Defensive security is the practice of protecting systems, detecting attacks in progress, and recovering from incidents. Core activities include:

  • Blue Teaming — continuous infrastructure monitoring, log analysis, and threat detection
  • Incident Response (IR) — structured process for containing, eradicating, and recovering from breaches
  • Threat Hunting — proactive, human-led investigation to detect hidden attacker presence
  • Digital Forensics — post-incident investigation to reconstruct attack timelines and preserve evidence
  • Security Awareness Training — educating employees to recognize phishing, social engineering, and policy violations

Real Example: A SIEM (Security Information and Event Management) system alerts a blue team analyst about 500 failed SSH login attempts from a single IP in 10 minutes. The analyst blocks the IP and investigates whether any logins succeeded.

The Six Stages of Penetration Testing

Every authorized penetration test follows a structured methodology. The phases are sequential but may loop iteratively as new information is discovered.

  1. 01
    Reconnaissance

    Information gathering — collecting as much data as possible about the target without triggering alerts. Techniques include OSINT (open-source intelligence), WHOIS lookups, DNS enumeration, Google dorking, social media profiling, and job listing analysis. This phase determines attack surface and entry points.

    Example: Checking LinkedIn to discover a target company uses Apache servers and AWS. Using WHOIS to find domain registration details. Using Google dorking to locate exposed configuration files.

  2. 02
    Scanning

    Active enumeration of the target network — identifying live hosts, open ports, running services, software versions, and potential vulnerabilities. Tools: Nmap, Netdiscover, Masscan, Nessus, OpenVAS. Banner grabbing reveals what software versions are exposed.

    Example: Nmap reveals port 22 (SSH) is open with an outdated OpenSSH version. Nessus confirms a known CVE against that version. This becomes a candidate for exploitation in phase 3.

  3. 03
    Gaining Access

    Exploiting discovered vulnerabilities to achieve unauthorized access. Techniques include exploiting known CVEs (via Metasploit or manually), password attacks (dictionary, brute-force, credential stuffing), phishing, and web application exploitation. This stage validates whether vulnerabilities are actually exploitable.

  4. 04
    Maintaining Access

    Establishing persistence to retain control over a compromised system across reboots and security tool actions. Techniques include deploying backdoors, creating new privileged accounts, installing rootkits, and privilege escalation to gain administrator or root access.

  5. 05
    Covering Tracks

    Minimizing forensic evidence of the attack to evade detection. Techniques: clearing authentication logs, disabling monitoring tools, hiding processes and files, timestomping (modifying file metadata), and using legitimate system utilities ("living off the land") to blend attack activity with normal operations.

  6. 06
    Reporting

    Documenting all findings in a structured report delivered to the client. Includes: executive summary (business risk impact), technical vulnerability details (CVE references, CVSS scores, proof-of-concept evidence), risk ratings (Critical/High/Medium/Low), and prioritized remediation recommendations.

❓ Likely Exam Questions — Week 1

  1. Q: What is the difference between offensive and defensive security? Give examples of each.

    Model Answer

    Offensive security proactively simulates attacks to find vulnerabilities before malicious actors do. Examples: penetration testing, red teaming, ethical hacking. Defensive security focuses on protecting systems and responding to attacks. Examples: firewalls, IDS/IPS, incident response, threat hunting, SIEM. Both disciplines are complementary — red team findings drive blue team improvements.

  2. Q: List and describe the six stages of a penetration test.

    Model Answer

    1. Reconnaissance — passive and active information gathering about the target. 2. Scanning — identifying open ports, services, and vulnerabilities. 3. Gaining Access — exploiting discovered vulnerabilities to enter the system. 4. Maintaining Access — establishing persistence through backdoors or elevated privileges. 5. Covering Tracks — deleting logs and hiding evidence of intrusion. 6. Reporting — documenting all findings with risk ratings and remediation recommendations.

  3. Q: What is the difference between a vulnerability and an exploit?

    Model Answer

    A vulnerability is a weakness or flaw in a system — for example, an unpatched version of Apache with a known buffer overflow bug. An exploit is the specific method, code, or technique that takes advantage of that vulnerability to achieve an unauthorized outcome. Every exploit targets a vulnerability, but not every vulnerability has a publicly available exploit.

  4. Q: What is Kali Linux and why is it used for penetration testing?

    Model Answer

    Kali Linux is a Debian-based Linux distribution maintained by Offensive Security specifically for penetration testing and digital forensics. It is used because it comes pre-installed with 600+ security tools (Nmap, Metasploit, Burp Suite, Aircrack-ng, etc.), is open-source, regularly updated, highly customizable, and has extensive community support. It eliminates setup overhead so testers can focus on testing.

Week 02

Information Gathering and OSINT

Instructor: Senior Lecturer, MSc Aliya Abdiraman

A. Lecture Overview

Reconnaissance is the most critical phase of any penetration test or real-world attack. This lecture covers both passive and active information gathering methodologies. Students learn to leverage Open Source Intelligence (OSINT) — the art of extracting actionable intelligence from publicly available sources — without ever directly touching the target system. Topics include WHOIS lookups, DNS enumeration, Google dorking, social media scraping, certificate transparency, file metadata analysis, and automated OSINT tools.

B. Core Concepts & Definitions

OSINT

Open Source Intelligence — the collection and analysis of information from publicly available sources (websites, social media, public records, DNS, WHOIS, certificate logs, job boards, etc.) without interacting directly with the target's systems.

Passive Reconnaissance

Information gathering that does not interact directly with the target system. Uses third-party databases and public data sources. The target has no way to detect this activity. Examples: WHOIS, Google dorking, Shodan queries, certificate transparency.

Active Reconnaissance

Information gathering that directly interacts with the target's systems — sending probes that may generate logs and trigger alerts. Examples: Nmap scans, banner grabbing, DNS zone transfer attempts, web spidering.

WHOIS

A protocol and database system that stores domain registration information — registrant name/organization, registrar, creation date, expiration date, name servers, and sometimes technical/admin contact emails. Essential for mapping ownership.

DNS Enumeration

The process of querying DNS records to discover subdomains, mail servers (MX), name servers (NS), IP addresses (A/AAAA), and text records (TXT). Tools: dig, nslookup, DNSRecon, Gobuster (DNS mode).

Google Dorking

Using advanced Google search operators to locate sensitive or misconfigured data exposed on the public internet — login pages, exposed files, database dumps, configuration files, and camera feeds. Also known as "Google hacking."

Certificate Transparency

A public log of all SSL/TLS certificates issued by Certificate Authorities (CAs). Attackers query these logs (via tools like crt.sh) to enumerate subdomains that a target owns — revealing infrastructure not listed in public DNS.

File Metadata (EXIF)

Exchangeable Image File Format data embedded in images, Office documents, and PDFs — revealing author name, software version, GPS coordinates, creation timestamps, and device information. Extractable with tools like ExifTool.

Social Media Scraping for OSINT

Attackers routinely mine LinkedIn, Twitter/X, Facebook, Instagram, and GitHub for intelligence. Key data gathered includes:

  • Technology stack — Job postings reveal what technologies a company uses ("3+ years React, AWS, PostgreSQL")
  • Key personnel — Names, roles, and contact information of executives and IT staff (targets for spear phishing)
  • Organizational structure — Department hierarchy useful for impersonation attacks
  • Network topology hints — Engineers posting about "our AWS us-east-1 infrastructure" or "our Kubernetes cluster"
  • Credentials — Developers accidentally committing API keys, passwords, and SSH private keys to public GitHub repositories

❓ Likely Exam Questions — Week 2

  1. Q: What is the difference between passive and active reconnaissance? Why does the distinction matter?

    Model Answer

    Passive reconnaissance gathers information without directly interacting with the target — using OSINT sources, WHOIS, Shodan, and certificate logs. It is nearly undetectable. Active reconnaissance directly probes target systems (Nmap scans, DNS zone transfers) and may generate logs and trigger IDS alerts. The distinction matters because passive recon carries minimal legal and detection risk, while active recon, if performed without authorization, may be illegal and can alert defenders to the ongoing assessment.

  2. Q: What is Google Dorking? Give three examples of dork operators and their security applications.

    Model Answer

    Google Dorking (Google Hacking) uses advanced search operators to find sensitive data exposed on public websites. Examples: (1) filetype:pdf confidential — finds exposed PDF files labeled confidential. (2) intitle:"index of" — finds web servers with directory listing enabled, potentially exposing files. (3) inurl:admin login — finds admin login pages. These techniques reveal misconfigurations and unintentional data exposure without interacting with the target directly.

Week 03

Network Scanning and Vulnerability Detection

Instructor: Senior Lecturer Aliya Abdiraman

A. Lecture Overview

Network scanning transitions reconnaissance from passive information gathering to active probing of target infrastructure. This lecture provides a deep technical grounding in network scanning concepts, port discovery, service fingerprinting, OS detection, and vulnerability scanning. Students master Nmap — the industry standard — as well as Zenmap (GUI) and Masscan (ultra-fast internet scanner). The lecture also covers IDS/IPS evasion techniques used to bypass security controls during assessments.

B. Core Concepts & Definitions

Network Scanning

The active process of discovering live hosts, open ports, running services, and potential vulnerabilities across a network. Used in both security assessments and network management. Leaves traces in target logs.

Port

A logical endpoint for network communication, numbered 0–65535. Well-known ports (0–1023) include 22 (SSH), 80 (HTTP), 443 (HTTPS), 21 (FTP), 25 (SMTP), 53 (DNS), 3306 (MySQL). Open ports indicate running services — and potential attack surfaces.

TCP SYN Scan (Half-Open)

Nmap's default stealth scan (-sS). Sends a SYN packet; if the target responds with SYN-ACK (port open), Nmap sends RST instead of ACK — never completing the handshake. Faster than full TCP and often not logged by applications.

TCP 3-Way Handshake

The connection establishment protocol: (1) SYN — client requests connection, (2) SYN-ACK — server acknowledges, (3) ACK — client confirms. Full open scans (-sT) complete this handshake and are more detectable.

Banner Grabbing

The technique of reading the service banner returned when connecting to an open port — revealing the software name, version, and sometimes OS. Enables targeted vulnerability lookup against known CVEs for that version.

NSE (Nmap Scripting Engine)

Lua-based scripting framework built into Nmap. Provides hundreds of scripts for vulnerability detection, authentication testing, brute-forcing, service enumeration, and exploitation. Key categories: auth, vuln, brute, exploit, fuzzer, broadcast.

IDS (Intrusion Detection System)

A security system that monitors network traffic or host activity for suspicious patterns and generates alerts. A passive system — it detects but does not block. Types: NIDS (network-based) and HIDS (host-based).

IPS (Intrusion Prevention System)

An active security system that monitors traffic AND can take automated action — blocking connections, dropping packets, or resetting sessions — when attack patterns are detected. IPS = IDS + active response.

TCP vs. UDP Comparison

TCP (Transmission Control Protocol)
  • Connection-oriented — requires 3-way handshake
  • Reliable — guarantees packet delivery and ordering
  • Slower due to acknowledgment overhead
  • Used by: HTTP, HTTPS, SSH, FTP, SMTP
  • Scannable — SYN responses indicate open ports clearly
UDP (User Datagram Protocol)
  • Connectionless — no handshake required
  • Unreliable — no delivery guarantee
  • Faster — minimal overhead
  • Used by: DNS (53), SNMP (161), DHCP (67/68), VoIP
  • Harder to scan — no response often means open OR filtered

❓ Likely Exam Questions — Week 3

  1. Q: Explain how a TCP SYN scan works and why it is considered "stealthy."

    Model Answer

    A TCP SYN scan (Nmap -sS) works by sending a SYN packet to the target port. If the port is open, the target responds with SYN-ACK. Instead of completing the 3-way handshake with an ACK, Nmap immediately sends a RST (reset) to tear down the half-open connection. This is considered stealthy because many applications only log fully established TCP connections — a connection that never completes the handshake may not be recorded in application logs, though network-level IDS can still detect the pattern.

  2. Q: What is the difference between an IDS and an IPS?

    Model Answer

    An IDS (Intrusion Detection System) is a passive system that monitors traffic and generates alerts when suspicious patterns are detected — it does not take action to stop traffic. An IPS (Intrusion Prevention System) is an active system that monitors traffic AND takes automated responses — dropping malicious packets, blocking source IPs, or resetting connections. IPS is positioned inline with traffic flow; IDS can operate out-of-band by analyzing a copy of traffic.

Week 04

Network Hacking and WPA/WPA2 Security

Instructor: Senior Lecturer Balziya Aldosh, School of Cybersecurity

A. Lecture Overview

Wireless networks represent a fundamentally different attack surface from wired infrastructure — they broadcast data through open air, accessible to any device within radio range. This lecture covers Wi-Fi security protocols from the now-broken WEP standard through WPA2 and the modern WPA3, analyzing the cryptographic weaknesses that make each protocol more or less resistant to attack. Students learn the Aircrack-ng toolkit for wireless auditing: capturing traffic with airodump-ng, executing deauthentication attacks with aireplay-ng, and cracking WEP/WPA2 credentials with aircrack-ng.

B. Core Concepts & Definitions

Wi-Fi Security Protocol Comparison

Protocol Encryption Security Level Crack Method
OPN None None All traffic visible in plaintext — passive sniffing captures credentials
WEP RC4 (weak IVs) Broken Statistical IV analysis — aircrack-ng cracks with ~60,000+ IVs (minutes)
WPA TKIP Deprecated Dictionary/brute-force against captured 4-way handshake
WPA2 AES-CCMP Strong (if long passphrase) Dictionary/brute-force against 4-way handshake — rockyou.txt etc.
WPA3 AES-GCMP-256 + SAE Very Strong Dragonblood vulnerabilities (niche); SAE blocks offline dictionary attacks

Monitor Mode

A wireless adapter operating mode that allows passive capture of ALL wireless frames in range — not just frames addressed to the adapter. Required for Wi-Fi packet capture. Enabled with: airmon-ng start wlan0

4-Way Handshake

The WPA/WPA2 authentication exchange between a client and access point — establishing the PTK (Pairwise Transient Key) used to encrypt data. This handshake contains cryptographic material that can be attacked offline to recover the passphrase if captured.

Deauthentication Attack

Sending forged 802.11 deauthentication frames to a connected client, forcing it to disconnect from the access point. When the client reconnects, the 4-way handshake can be captured. Tool: aireplay-ng -0 20 -a [BSSID] wlan0mon

Initialization Vector (IV)

A random value used with WEP encryption to vary the keystream. WEP's fatal weakness: the IV space is only 24 bits (16.7 million values), and reused IVs allow statistical reconstruction of the encryption key. WPA/WPA2 fixed this with TKIP/CCMP.

SAE (Simultaneous Authentication of Equals)

WPA3's "Dragonfly handshake" — replaces the pre-shared key (PSK) model of WPA2. SAE prevents offline dictionary attacks because each authentication attempt requires active interaction with the access point, making brute-force impractical at scale.

BSSID / ESSID

BSSID (Basic Service Set Identifier) = the MAC address of the access point. ESSID (Extended SSID) = the human-readable Wi-Fi network name. Required parameters when targeting a specific access point during wireless assessment.

Cookie headers containing PHPSESSID session tokens are also visible and can be reused to hijack authenticated sessions — this is called session hijacking.

❓ Likely Exam Questions — Week 4

  1. Q: Why is WEP considered completely broken despite using encryption?

    Model Answer

    WEP is broken because of fundamental flaws in how it uses RC4 encryption with Initialization Vectors (IVs). The IV is only 24 bits, meaning only ~16.7 million unique values exist — in a busy network, IVs repeat quickly. Repeated IVs produce the same keystream, which allows statistical analysis to recover the encryption key. Using tools like aircrack-ng, with approximately 60,000 captured IVs, the key can be recovered in minutes regardless of its length.

  2. Q: What is a deauthentication attack and why is it used in WPA2 cracking?

    Model Answer

    A deauthentication attack involves sending forged 802.11 deauthentication management frames to force connected clients off an access point. When the client reconnects, it must perform the WPA2 4-way handshake with the access point. An attacker capturing this handshake can then perform an offline dictionary or brute-force attack against the cryptographic material to try to recover the passphrase — without needing to be authenticated to the network.

Week 05

Social Engineering Techniques

Instructor: Senior Lecturer Aliya Abdiraman

A. Lecture Overview

Social engineering is the art of manipulating people into revealing confidential information or performing actions that compromise security — bypassing technical controls by targeting the human element. This lecture explores the psychological principles underpinning social engineering attacks, the taxonomy of attack types (phishing, spear phishing, whaling, vishing, smishing, pretexting, tailgating), and the tools attackers use. A critical component is the defensive perspective: building organizational security awareness cultures that resist social manipulation.

B. Core Concepts & Definitions

Social Engineering

Psychological manipulation of individuals into performing actions or divulging information that benefits an attacker. Exploits human cognitive biases — trust, fear, urgency, authority — rather than technical vulnerabilities.

Pretexting

Creating a fabricated scenario (pretext) to deceive a target into providing information or access. Example: impersonating IT support ("I'm from the helpdesk — I need your password to fix your account") or a vendor auditor.

Phishing

Mass-distributed fraudulent communications (usually email) impersonating trusted entities to harvest credentials or deliver malware. Success rate is statistical — even a 1% click rate on 10,000 emails is 100 compromised targets.

Spear Phishing

Targeted phishing crafted using personal intelligence about the victim — name, employer, role, colleagues, recent activities. Far more convincing and effective than generic phishing. Used in APT campaigns.

Whaling

Spear phishing targeting high-value individuals — CEOs, CFOs, board members. Often used for Business Email Compromise (BEC) — fraudulently authorizing large financial transfers.

Vishing

Voice phishing — using phone calls to manipulate targets. Attackers spoof caller IDs to impersonate banks, IT support, government agencies. Tools like SpoofApp, SpoofCard, or Asterisk VoIP make caller ID falsification trivial.

Tailgating / Piggybacking

Physical social engineering — following an authorized person through a secured door without using credentials. Relies on social norms of politeness (holding doors open). Countered by mantraps and security awareness training.

BadPDF Attack

A social engineering technique where a maliciously crafted PDF is sent to a target. The PDF contains embedded code or forces an external resource request that can capture the target's credentials (NTLMv2 hash) or install malware when opened.

C. Expanded Academic Notes — Methods of Influence

Robert Cialdini's principles of influence are the psychological toolkit of social engineers. Understanding them is essential for both mounting and defending against social engineering attacks.

👑 Authority

Targets comply with perceived authority figures — impersonating executives, IT administrators, law enforcement, or regulatory bodies. "I'm from corporate IT — I need your login to apply a security patch."

⏰ Urgency / Scarcity

Creating artificial time pressure eliminates careful deliberation. "Your account will be locked in 10 minutes — verify your credentials now." Urgency overrides rational evaluation of suspicious requests.

👥 Social Proof

Exploiting the tendency to follow the crowd. "Everyone else in your department has already updated their password through this link." Effective in unfamiliar or confusing situations where targets look to others for behavioral cues.

💗 Likeness

Targets are more likely to comply with people they like or relate to. Attackers research targets' interests, hobbies, and contacts — then impersonate or reference shared connections.

😱 Fear

Fear-based manipulation creates panic that bypasses rational decision-making. "Your computer has been infected with a virus — call this number immediately." Used in tech support scams and ransomware social engineering.

Famous Social Engineering Incidents

Twitter Bitcoin Hack (2020)

Attackers social-engineered Twitter employees into providing access to internal admin tools by impersonating IT support. Result: 130 high-profile accounts (Obama, Gates, Musk, Apple) hijacked for a Bitcoin scam netting ~$120,000 within hours.

RSA SecurID Breach (2011)

A targeted spear phishing email with an Excel spreadsheet titled "2011 Recruitment Plan.xls" compromised RSA employees — eventually leading to theft of seed values for millions of SecurID tokens used by defense contractors.

Kevin Mitnick — The Art of Deception

The most prolific social engineer in history. Mitnick compromised major corporations primarily through phone-based pretexting — impersonating employees, IT support, and vendors to extract passwords and access credentials without any technical hacking.

BeEF (Browser Exploitation Framework)

Client-Side

SpoofCard / SpoofApp

Vishing

iOS/Android apps for caller ID spoofing — change displayed number, alter voice, record calls, generate background noise. Used to make vishing calls appear to come from legitimate phone numbers.

    Model Answer

    Authority — targets comply with perceived authority figures (impersonating executives, IT, law enforcement). Urgency/Scarcity — artificial time pressure prevents careful evaluation ("act now or your account is locked"). Social Proof — exploiting the tendency to follow others in unfamiliar situations.   2. Q: What is the difference between phishing, spear phishing, and whaling?

Week 06

Web Application Security

Instructor: Assistant Professor, PhD Akzhibek Amirova

A. Lecture Overview

Web applications are the dominant attack surface in modern enterprise environments. This lecture provides a structured introduction to web application security — from foundational security models (CIA Triad, CVSS, OWASP Top 10) through a taxonomy of common vulnerability classes. Students learn to use Burp Suite as an intercepting proxy and explore both server-side vulnerabilities (SQL injection, command injection, LFI, SSRF, path traversal) and client-side vulnerabilities (XSS, CSRF, session fixation). Practical labs use DVWA (Damn Vulnerable Web Application) and WebGoat.

B. Core Concepts & Definitions

The CIA Triad

🔒 Confidentiality

Information is not disclosed to unauthorized individuals, entities, or processes. Breached by: data exfiltration, SQL injection, path traversal, credential theft.

✅ Integrity

Data maintains accuracy and completeness throughout its lifecycle. Breached by: unauthorized database modifications, parameter tampering, man-in-the-middle attacks.

📈 Availability

Systems and data are accessible to authorized entities on demand. Breached by: DDoS attacks, ransomware, resource exhaustion vulnerabilities.

OWASP Top 10

The Open Web Application Security Project's authoritative list of the ten most critical web application security risks — updated periodically based on data from thousands of real-world assessments. The definitive baseline for web security testing.

CVSS

Common Vulnerability Scoring System — provides a standardized numerical score (0–10) reflecting vulnerability severity based on attack vector, complexity, privileges required, user interaction, and impact on CIA. Calculator at first.org/cvss.

Command Injection

A vulnerability where attacker-controlled input is passed to OS shell commands without sanitization — allowing arbitrary system command execution on the server. Example: system("ping -c 1 " + ip) — if ip is 127.0.0.1;id, the id command also executes.

Path Traversal

A vulnerability where an attacker manipulates file path parameters to access files outside the intended directory. Classic payload: ../../../../../etc/passwd. Can expose sensitive system files, configuration files, and credentials.

XSS (Cross-Site Scripting)

A vulnerability where attacker-controlled HTML/JavaScript is injected into web pages and rendered in victims' browsers — enabling cookie theft, session hijacking, keylogging, and browser-based attacks. Payload example: <script>alert(document.cookie)</script>

XXE (XML External Entity)

An attack against weakly configured XML parsers — injecting external entity declarations that force the server to fetch internal files or make outbound network requests, enabling SSRF, file disclosure, and DoS.

JWT Attacks

JSON Web Tokens (format: base64.base64.base64) are used for authentication. Attacks include: none algorithm bypass, weak secret brute-force, algorithm confusion (RS256 to HS256), and key injection — allowing attackers to forge authentication tokens.

Session Fixation

An attacker sets a victim's session ID before authentication — then after the victim logs in, the attacker can use the known session ID to access the account. Prevented by regenerating session IDs upon authentication.

Brute Force Attack

Systematically attempting all possible values for credentials, encryption keys, or hidden resources. Countered by account lockout policies, CAPTCHA, rate limiting, and multi-factor authentication.

Server-Side vs Client-Side Vulnerabilities

Server-side vulnerabilities execute payloads on the server (SQL injection, command injection, LFI, XXE, SSRF — impact: database access, RCE, file disclosure). Client-side vulnerabilities execute in the victim's browser (XSS, CSRF — impact: session theft, unauthorized actions).

  1. Q: What is the CIA Triad? Give an example of a web attack that violates each component.

    Model Answer

    Confidentiality (information not disclosed to unauthorized parties) — violated by SQL injection extracting the users table. Integrity (data remains accurate and unmodified) — violated by a stored XSS attack injecting malicious scripts into a database that other users then see. Availability (systems accessible when needed) — violated by a DDoS attack overwhelming a web server. All three are essential; a secure system must preserve all three simultaneously.

  2. Q: What is Cross-Site Scripting (XSS)? Describe the difference between reflected and stored XSS.

    Model Answer

    XSS is a vulnerability where attacker-controlled JavaScript is injected into web pages rendered in victim browsers. Reflected XSS: the malicious script is included in the URL — the victim must click the crafted link. The script executes once and is not stored on the server. Stored XSS: the malicious script is stored on the server (in a database) and executes for every user who views the infected page — without requiring them to click a special link. Stored XSS is more dangerous due to its persistent and automatic nature.

Week 07

Web Application Attacks and Vulnerability Assessment Tools

Instructor: Assistant Professor, PhD Akzhibek Amirova

A. Lecture Overview

This lecture bridges the gap between understanding web vulnerabilities theoretically and finding them systematically at scale using automated tools. Students learn the critical distinction between SAST and DAST testing methodologies — and the toolsets that implement each approach. Key tools covered include SonarQube and Graudit (SAST), and OWASP ZAP, OpenVAS/Greenbone, and Nikto (DAST). Students also learn defensive countermeasures including WAFs, YARA rules, and backdoor scanning.

B. Core Concepts & Definitions

SAST (Static Application Security Testing)

White-box method — analyzes source code, bytecode, or binaries without executing the application. Finds vulnerabilities early in the development lifecycle (shift-left security). Requires access to source code.

  • Analyzes code structure, data flow, and control flow
  • Can identify: SQL injection patterns, hardcoded credentials, insecure crypto, input validation flaws
  • No false negatives from runtime state
  • Tools: SonarQube, Graudit, Checkmarx, Semgrep

DAST (Dynamic Application Security Testing)

Black-box method — tests a running application by sending malformed inputs and analyzing responses. No source code access required. Simulates real attacker behavior.

  • Tests authentication, session management, input handling at runtime
  • Finds: runtime-dependent vulnerabilities, misconfigurations, authentication flaws
  • May miss vulnerabilities that require specific code paths
  • Tools: OWASP ZAP, Nikto, OpenVAS, Burp Suite Scanner

WAF (Web Application Firewall)

A security device (hardware or software) that filters, monitors, and blocks HTTP/S traffic to and from web applications based on rules detecting attack patterns — SQL injection signatures, XSS payloads, path traversal attempts. A detection/delay layer, not a substitute for secure code.

YARA Rules

A pattern-matching language for malware identification. YARA rules define patterns (strings, byte sequences, conditions) that identify malware families, webshells, or malicious code in files. Widely used by AV vendors and threat intelligence teams. Helped identify encrypted Cobalt Strike payloads.

Week 08

Website Hacking – SQL Injection Vulnerabilities

Instructor: Senior Lecturer Balziya Aldosh, School of Cybersecurity

A. Lecture Overview

SQL Injection remains one of the oldest, most pervasive, and most catastrophic web application vulnerability classes — consistently featured in every OWASP Top 10 since its inception. This lecture provides a deep foundation in both SQL fundamentals and the mechanics of injection attacks. Students learn how relational databases work, the types of SQL commands, how injection vulnerabilities arise from improper input handling, and how attackers exploit them — from basic WHERE clause bypass to UNION-based data extraction. Hands-on labs use PortSwigger Web Academy and Burp Suite.

B. Core Concepts & Definitions

SQL (Structured Query Language)

The standard language for managing relational databases — querying, inserting, updating, deleting data, and controlling access. Developed by IBM (Chamberlin & Boyce) in the 1970s. Standardized by ANSI (1986) and ISO (1987). Dialects include T-SQL (SQL Server) and PL/SQL (Oracle).

SQL Injection (SQLi)

A vulnerability class where attacker-controlled input is concatenated with SQL queries — allowing manipulation of query logic, data extraction, authentication bypass, and (in some configurations) operating system command execution.

UNION-Based SQLi

Uses the SQL UNION operator to append additional SELECT statements to the original query — allowing extraction of data from arbitrary tables. Requires matching the column count and data types of the original query.

Boolean-Based Blind SQLi

When the application returns different responses for true vs. false conditions — but does not display database content directly. Attackers infer data character-by-character by observing behavioral differences in responses.

Time-Based Blind SQLi

Infers data from application response timing — injecting conditional delay functions (SLEEP(), WAITFOR DELAY). If the application pauses when a condition is true, data can be extracted one bit at a time.

Error-Based SQLi

Exploits verbose database error messages that include extracted data. The injected query is crafted to cause a database error that contains the value of interest in the error message displayed to the attacker.

SQL Command Categories

Category Abbreviation Commands Purpose
Data Query LanguageDQLSELECTRetrieve data from tables
Data Manipulation LanguageDMLINSERT, UPDATE, DELETE, JOINAdd, modify, and remove data
Data Definition LanguageDDLCREATE, ALTER, DROP, TRUNCATEDefine and modify table structure

C. Expanded Academic Notes

How SQL Injection Works — Step by Step

The fundamental principle: user-controlled data is concatenated with SQL query strings, causing the database to interpret attacker-supplied input as SQL commands rather than data values.

Vulnerable Code Pattern:

query = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"

Normal Input: username = alice, password = secret123

SELECT * FROM users WHERE username='alice' AND password='secret123'

Attack Input: username = admin'--, password = anything

SELECT * FROM users WHERE username='admin'-- AND password='anything'

The -- comments out the rest of the query. The password check is eliminated. Attacker logs in as admin without knowing the password.

UNION Attack — Extracting Data

Step 1: Determine column count

' UNION SELECT 'abc','def'--

If two columns exist, "abc" and "def" appear in output. If error, adjust column count.

Step 2: Extract database version (MySQL)

' UNION SELECT @@version, NULL--

Step 3: Extract database version (Oracle)

' UNION SELECT banner, NULL FROM v$version--

Historical SQLi Breaches

Heartland Payment Systems (2007–2008)

SQL injection compromised 130+ million credit/debit card numbers. At the time, the largest known data breach in US history. Total cost exceeded $140 million.

NASA (2009)

SQL injection exposed administrative credentials of 25 employees at NASA's Instrument Systems and Technology Division — demonstrating even critical infrastructure organizations are vulnerable.

Wall Street Journal (2014)

Attacker gained database access and offered to sell the newspaper's reader data online, demonstrating the monetization of SQLi breaches beyond immediate credential theft.

Testing for SQL Injection — Basic Approach

The single rule for detecting SQLi: trigger anomalies by sending unexpected data. Test all input points — GET parameters, POST body, cookies, HTTP headers, path parameters.

  • Append ' (single quote) — causes database syntax error if vulnerable
  • Append '' (two single quotes) — fixes the error (value still treated as data)
  • Boolean tests: ' OR '1'='1 (always true) vs ' OR '1'='2 (always false)
  • Time delays: '; SLEEP(5)-- — if page takes 5+ seconds, time-based blind SQLi likely exists

❓ Likely Exam Questions — Week 8

  1. Q: Explain the concept of SQL injection. How does an attacker bypass authentication using SQLi?

    Model Answer

    SQL injection occurs when user-controlled input is concatenated into SQL queries without parameterization, allowing the input to be interpreted as SQL code. For authentication bypass: if the login query is SELECT * FROM users WHERE username='[input]' AND password='[input]', injecting admin'-- as the username produces SELECT * FROM users WHERE username='admin'--' AND password='anything'. The -- comments out the password check, returning the admin user without password verification.

  2. Q: What is a UNION-based SQL injection attack? What conditions must be met for it to work?

    Model Answer

    A UNION-based attack appends an additional SELECT statement to the original query using the UNION operator — extracting data from arbitrary tables. For UNION to work: (1) the number of columns in the injected SELECT must match the original query, (2) the data types of corresponding columns must be compatible, and (3) the application must display the additional results. Attackers determine column count by incrementally adding NULL columns until no error occurs, then replace NULLs with data-extracting expressions.

Week 09

Advanced SQL Injection and Database Exploitation Concepts

Instructor: Senior Lecturer Balziya Aldosh, School of Cybersecurity

A. Lecture Overview

This lecture advances from basic SQL injection theory to professional-grade code review methodology and defense architecture. Students learn to read application code and systematically identify every user-controlled input source that could reach a database query. The lecture establishes a severity impact model (Critical/High/Medium/Low) based on query location, affected tables, and authentication requirements. Defense sections cover parameterized queries, input whitelisting, least-privilege database accounts, centralized query layers, WAF deployment, and database hardening — presenting the complete picture from attack to remediation.

B. Core Concepts & Definitions

Parameterized Queries

The primary defense against SQL injection. Query structure and user data are sent to the database separately — the database compiles the query first, then slots in the values. Values cannot be interpreted as SQL commands regardless of content. Also called "prepared statements."

Input Whitelisting

For cases where parameterization is impossible (column names, table names, ORDER BY direction) — define an allowlist of acceptable values in code. Any input not on the list is rejected or replaced with a safe default.

Least Privilege (Database)

The application's database account should have only the minimum permissions required for its legitimate functionality. A read-only app shouldn't have INSERT/UPDATE/DROP. Limits blast radius of successful injection.

Centralized Query Layer

A single codebase module through which all database interactions are routed — enforcing parameterization structurally. Makes it hard (or impossible) to write unsafe queries, eliminating developer inconsistency as a vulnerability source.

SQLi Severity Impact Model

Severity Conditions Example
CRITICALUnauthenticated + touches credentials/auth logic/login — any attacker can bypass authentication, dump passwords
HIGHAuthenticated + touches sensitive data (payments, secrets)/orders — authenticated user exposes all users' financial records
MEDIUMAuthenticated + touches application data (products, content)/search — injects into products table (limited sensitivity)

User-Controlled Input Sources — Code Review Checklist

When reviewing code for SQL injection, trace ALL input sources — not just obvious form fields:

  • URL query string — request.args.get('id') from /product?id=1
  • POST body — request.form.get('username') from login forms
  • URL path parameters — /user/3 where "3" is extracted and used in query
  • HTTP Headers — User-Agent (logged to DB), X-Forwarded-For (IP logging), Accept-Language (localization queries)
  • Cookies — session_id (session lookup), user_pref (theme queries), last_search (search functionality)

C. Expanded Academic Notes — Code-Level Defenses

❌ Vulnerable Pattern

query = "SELECT * FROM users WHERE username='" + username + "'"

Attack: username = admin'--

SELECT * FROM users WHERE username='admin'-- [password check gone]

✅ Parameterized Query

cursor.execute("SELECT * FROM users WHERE username = ?", (username,))

Attack: username = admin'--

DB searches literally for user named "admin'--" → finds nothing → login fails

Least Privilege Defense in Action

Even if injection succeeds, least privilege limits attacker actions:

  • Attacker tries: ' UNION SELECT * FROM admin_keys-- → Result: permission denied (app_user has no SELECT on admin_keys)
  • Attacker tries: '; DROP TABLE users-- → Result: permission denied (app_user has no DROP privilege)
  • Attacker tries: '; INSERT INTO users VALUES ('hacker','pass','admin')-- → Blocked if app is read-only

Platform-Level Defenses

Database Hardening

Disable features not needed by the application: xp_cmdshell (SQL Server), LOAD_FILE, INTO OUTFILE (MySQL), UTL_HTTP (Oracle). Reduce default attack surface.

Web Application Firewall (WAF)

Inspects every HTTP request for known SQLi patterns before it reaches application code. Blocks automated scanners and script kiddies. Not a substitute for parameterized queries — skilled attackers bypass WAFs regularly.

Network Isolation

Database server in a private network segment — accessible only from application servers, not from the internet. Prevents attackers from pivoting to the database server even after code injection.

Error Handling

Never display verbose database errors to users. Log errors server-side. Generic error messages ("Something went wrong") prevent error-based SQLi and hide database structure from attackers.

❓ Likely Exam Questions — Week 9

  1. Q: Why are parameterized queries the primary defense against SQL injection? How do they work?

    Model Answer

    Parameterized queries work by separating the SQL command structure from the data values. The application sends the query template to the database first (e.g., SELECT * FROM users WHERE username = ?), which is compiled. The values are then sent separately and slotted into the pre-compiled structure. Because the structure is already compiled, the database cannot interpret the values as SQL commands — a payload like admin'-- is searched for literally as a username string, not interpreted as SQL. This separation makes injection structurally impossible regardless of what the attacker provides as input.

  2. Q: You find a SQLi vulnerability in a /search route that queries the products table and requires authentication. What severity would you assign and why?

    Model Answer

    MEDIUM severity. Reasoning: (1) Authentication is required — reduces attack surface to authenticated users only, not all internet attackers. (2) The affected table is products — a low-sensitivity table containing no credentials, personal data, or financial records. (3) Impact is limited to dumping product information that may already be partially public. If the same vulnerability existed without authentication or in a table containing credentials or payment data, severity would increase to HIGH or CRITICAL respectively.

Master Glossary

APT
Advanced Persistent Threat — a sophisticated, prolonged cyberattack by well-resourced actors (often nation-state) targeting specific organizations
BEC
Business Email Compromise — fraudulent emails impersonating executives to authorize financial transfers
BSSID
Basic Service Set Identifier — the MAC address of a wireless access point
CAPTCHA
Completely Automated Public Turing test to tell Computers and Humans Apart — prevents automated brute-force attacks
CIA Triad
Confidentiality, Integrity, Availability — the three core pillars of information security
CVE
Common Vulnerabilities and Exposures — standardized identifier for publicly known security vulnerabilities (e.g., CVE-2021-44228)
CVSS
Common Vulnerability Scoring System — numerical (0–10) severity rating for vulnerabilities
DAST
Dynamic Application Security Testing — black-box testing of running applications
DNS
Domain Name System — translates human-readable domain names to IP addresses
ESSID
Extended Service Set Identifier — the human-readable Wi-Fi network name (SSID)
Exploit
A technique or code that takes advantage of a vulnerability to achieve an unauthorized effect
IDS
Intrusion Detection System — passive system that detects and alerts on suspicious activity
IPS
Intrusion Prevention System — active system that detects AND blocks suspicious activity
IV
Initialization Vector — a random value used with encryption; WEP's 24-bit IVs are its fatal weakness
JWT
JSON Web Token — compact, URL-safe authentication token (format: header.payload.signature)
MFA
Multi-Factor Authentication — requires two or more verification methods before granting access
NSE
Nmap Scripting Engine — Lua-based scripting framework for extending Nmap with custom scripts
OSINT
Open Source Intelligence — intelligence collected from publicly available sources
OWASP
Open Web Application Security Project — non-profit producing the OWASP Top 10 and security guides
Parameterized Query
SQL query where structure and user data are sent separately — the primary defense against SQL injection
Payload
The malicious code or data component of an exploit — what is delivered after exploitation
Phishing
Mass-distributed fraudulent communications impersonating trusted entities to harvest credentials or deliver malware
Privilege Escalation
Gaining higher levels of access than initially obtained — from user to admin, or from app to OS
SAE
Simultaneous Authentication of Equals — WPA3's "Dragonfly" authentication protocol replacing WPA2's PSK
SAST
Static Application Security Testing — white-box code analysis without executing the application
SIEM
Security Information and Event Management — aggregates and correlates security logs for threat detection
SQL Injection
Vulnerability where attacker-controlled input manipulates database queries — enabling data theft, authentication bypass, and (sometimes) RCE
SSRF
Server-Side Request Forgery — exploiting a server to make HTTP requests to internal/external resources on behalf of the attacker
Threat Hunting
Proactive, human-led investigation to detect attacker presence not caught by automated systems
Vulnerability
A weakness or flaw in a system that could be exploited to compromise security
WAF
Web Application Firewall — HTTP traffic filter that blocks known attack patterns
WHOIS
Protocol/database for querying domain registration information — ownership, registrar, name servers
XSS
Cross-Site Scripting — injection of malicious JavaScript into web pages rendered in victim browsers
XXE
XML External Entity — attack against weakly configured XML parsers enabling file disclosure and SSRF
YARA
Pattern-matching language for malware and threat identification in files
Zero-Day
A vulnerability unknown to the vendor — no patch exists; often used in sophisticated targeted attacks
